## 2. OCR Implementation

### Files involved

| File | Role |
|---|---|
| [`src/lib/ocr/ocr.ts`](src/lib/ocr/ocr.ts) | Canvas crop, ink check, preprocessing — **legacy Tesseract.js code still present** |
| [`src/lib/api/manga-ocr-py-API.ts`](src/lib/api/manga-ocr-py-API.ts) | HTTP client for py-mekai-api (OCR + translate) |
| [`src/lib/utils/browserAPI.ts`](src/lib/utils/browserAPI.ts) | Main OCR+translate orchestrator (`ocrAndTranslate()`) |
| [`src/ui/components/OCRSelectionLayer.tsx`](src/ui/components/OCRSelectionLayer.tsx) | Mouse drag UI to select a page region |
| [`src/ui/pages/MangaReaderPage.tsx`](src/ui/pages/MangaReaderPage.tsx) | Calls `ocrAndTranslate()`, saves result, renders overlays |
| [`py-mekai-api/main.py`](py-mekai-api/main.py) | Python FastAPI server — `/ocr` (PaddleOCR) + `/translate` (OPUS-MT) |

### Current active OCR flow

```
User drag-selects a region
  → OCRSelectionLayer.onSelect()
  → MangaReaderPage.handlePageSelect()
  → browserAPI.ocrAndTranslate(imgEl, bbox)
      ├─ ocr.ts/hasInkContent()      ← pre-flight Otsu binarization check
      ├─ isMangaOcrAvailable()        ← probe /ocr/health (30 s cache)
      ├─ ocr.ts/cropToDataUrl()       ← crop + base64 encode region
      ├─ localMangaOcr(base64)        ← POST /ocr → PaddleOCR Python model
      ├─ translateJapaneseToEnglishWithProvider()
      │    └─ localTranslateJaToEn()  ← POST /translate → OPUS-MT
      └─ toRomaji()                   ← client-side romanisation
  → save to translation_history (private)
  → if translator → upsert to chapter_translations (published)
  → render TranslationOverlay
```

### Legacy OCR code still present in `ocr.ts`

[`src/lib/ocr/ocr.ts`](src/lib/ocr/ocr.ts) still exports and defines:
- [`ocrFromImageElement()`](src/lib/ocr/ocr.ts:444) — full Tesseract.js pipeline with `createWorker`, PSM 6 / PSM 11 retry
- Internal helpers: `cropToCanvas()`, `preprocessCanvasForManga()` (Otsu), `tightenToInk()`, `rotateIfVertical()`, `labelComponents()`, `cleanText()`

Of those, only **two helpers are actually consumed** by the active pipeline via `src/lib/utils/browserAPI.ts`:
- [`hasInkContent()`](src/lib/ocr/ocr.ts:376) — pre-flight ink check (uses `cropToCanvas` + `preprocessCanvasForManga` internally)
- [`cropToDataUrl()`](src/lib/ocr/ocr.ts:416) — crop region to base64 PNG (uses `cropToCanvas`)

The main `ocrFromImageElement()` function (and all its supporting internal code) is **unused** by the active code path; it is dead Tesseract.js code.

## 5. Duplicate Logic

### Issue 1 — Duplicate translation storage path

When a translator performs OCR, **two separate DB writes** happen in sequence inside `MangaReaderPage.handlePageSelect()`:

1. `addHistory.mutateAsync(...)` → writes to `translation_history`
2. `upsertChapterTranslation(...)` → writes to `chapter_translations`

Both payloads contain nearly identical fields (`chapter_id`, `page_index`, `region`, `region_hash`, `ocr_text`, `translated`, `romaji`). This is intentional by design (private vs published) but the construction of each payload is inline in the page component, which is fragile.

**Suggestion:** Extract a `saveTranslation(chapter, pageIndex, region, ocrResult, canPublish)` service function that handles both writes atomically and is reusable.

### Issue 2 — Duplicate `regionHash` computation

`regionHash(sel.region)` is called **twice** in `handlePageSelect()` — once for `addHistory` and once for `upsertChapterTranslation`. Though cheap, it should be computed once and reused.

### Issue 3 — `addHistoryEntry` / `deleteHistoryEntry` alias exports

[`src/services/translationHistory.ts`](src/services/translationHistory.ts) exports both:
- `createTranslationHistory` (canonical)
- `addHistoryEntry` (alias)
- `deleteTranslationHistory` (canonical)
- `deleteHistoryEntry` (alias)

These two aliases create confusion. Recommend removing them and using canonical names everywhere.

### Issue 4 — `hasInkContent` duplicates inner pixel-scan work

[`hasInkContent()`](src/lib/ocr/ocr.ts:376) re-runs `cropToCanvas()` + `preprocessCanvasForManga()` + a pixel scan. Then `cropToDataUrl()` (called afterward) re-runs `cropToCanvas()` again. The same canvas crop is computed **twice** per OCR call. A shared `prepareCanvas()` helper returning a cached canvas would eliminate this.

## 6. Reader Overlay System

### Rendering pipeline

1. `MangaReaderPage` maintains `userOverlays` (from `translation_history`) and `publishedOverlays` (from `chapter_translations`).
2. Both are merged via `useMemo` into `mergedOverlays`, deduped by `key = "${pageIndex}-${regionHash}"`. User overlays take priority over published.
3. Per page: `ReaderPageItem` receives only the overlays for its `pageIndex`.
4. Each overlay is rendered by [`TranslationOverlay`](src/ui/components/TranslationOverlay.tsx) using `position: absolute` with percentage-based positioning:
   ```
   left:   region.x * 100%
   top:    region.y * 100%
   width:  region.w * 100%
   height: region.h * 100%
   ```
5. Text is rendered onto a `<canvas>` using `drawFittedText()` — shrink-to-fit with word wrapping, minimum font size guard, and truncation with ellipsis.

### Coordinate system

Overlays use the `RegionBox { x, y, w, h }` type where all values are **fractions 0–1** of the image dimensions. This is correct and consistent across:
- `OCRSelectionLayer` (computes fractional coords from mouse/image rect)
- `TranslationOverlay` (renders with percentage CSS)
- `chapter_translations.region` JSONB column
- `translation_history.region` JSONB column
- `regionHash()` in `types/index.ts`

**No coordinate mismatch detected.**

### ResizeObserver re-render

`TranslationOverlay` uses a `ResizeObserver` to re-render the canvas when the overlay is resized (e.g. window resize). This is correct but creates **one ResizeObserver per overlay**. With many overlays open this could accumulate observers. Consider a single shared ResizeObserver on the page container.

## 7. Performance Risks

| Risk | Location | Severity |
|---|---|---|
| **Double canvas crop per OCR call** | `src/lib/utils/browserAPI.ts` calls `hasInkContent()` then `cropToDataUrl()` — `cropToCanvas()` runs twice | Medium |
| **One Tesseract Worker created and destroyed per OCR call** | `src/lib/ocr/ocr.ts` — `ocrFromImageElement()` | High (but function is currently unused) |
| **Whole CBZ fetched and unzipped in browser memory** | `MangaReaderPage` CBZ extraction via JSZip | Medium — large chapters (50+ pages) will stress RAM |
| **All page images extracted at once as Object URLs** | `Promise.all(imageEntries.map(...))` | Medium — no lazy/progressive loading |
| **One ResizeObserver per TranslationOverlay** | `TranslationOverlay.tsx` | Low-Medium at scale |
| **`supabase.auth.getUser()` called on every DB write** | `chapterTranslations.ts`, `translationHistory.ts` | Low — cached by SDK but still async overhead |
| **Service availability probed on every `ocrAndTranslate()` call** | `src/lib/api/manga-ocr-py-API.ts` | Low — 30 s cache mitigates, but first call in window has latency |
| **Scroll-mode reading progress writes every 2 s** | `MangaReaderPage` IntersectionObserver debounce | Low |

## 9. Recommended Architecture Changes

### Current architecture summary

```
User selects region
  → [OCRSelectionLayer] drag UI
  → [browserAPI.ts] ocrAndTranslate()
       ├─ [ocr.ts] hasInkContent() + cropToDataUrl()    ← keep, refactor
       ├─ [manga-ocr-py-API.ts] localMangaOcr()         ← keep
       └─ [translate.ts] translateJapanese...()         ← keep
  → [MangaReaderPage] save to DB + render overlay
```

### Proposed improvements

#### A — Trim `ocr.ts` to only what is used

Remove all dead Tesseract.js code (see §8). Rename the file to `imageUtils.ts` or `cropUtils.ts` to reflect its actual role: image cropping and ink pre-flight check. This removes the `tesseract.js` dependency entirely.

#### B — Eliminate double canvas crop

Refactor `browserAPI.ocrAndTranslate()`:
```
const canvas = cropToCanvas(imgEl, bbox)
preprocessCanvasForManga(canvas)
if (!hasInkOnCanvas(canvas)) return empty result
const base64 = canvas.toDataURL('image/png')
const ocrText = await localMangaOcr(base64)
```
This runs `cropToCanvas` + `preprocessCanvasForManga` only once.

#### C — Centralise the save-translation logic

Extract `saveTranslationResult(chapter, pageIndex, region, ocrResult, { publishIfTranslator })` from `MangaReaderPage.handlePageSelect()` into a service or hook. This makes the double-write (history + published) explicit, testable, and reusable.

#### D — Add an error boundary for OCR service unavailability

Currently if `py-mekai-api` is offline, the user sees a raw error string. Add a detected "service offline" state that renders a helpful setup guide (already partially done via `isMangaOcrAvailable()`).

#### E — Future: Auto-detect speech bubbles (no user selection needed)

The current UX requires manual region selection. The new architecture described in the task specification targets automatic detection:

```
Chapter page loaded
  → Speech bubble detection (Python: OpenCV contour / YOLOv8 text region model)
       POST /detect → [{ x, y, w, h }, ...]
  → For each region: localMangaOcr() + localTranslateJaToEn()
  → Bulk upsert to chapter_translations
  → TranslationOverlay renders all regions
```

Integration point in the frontend: after CBZ extraction and before first render, trigger `POST /detect` with each page image and pre-populate `publishedOverlays`. A new `/detect` endpoint must be added to `py-mekai-api/main.py`.

#### F — Consider lazy CBZ page extraction

Instead of extracting all pages at once with `Promise.all`, extract the first 3 pages immediately and extract remaining pages on demand (IntersectionObserver approach). This reduces initial memory pressure for large chapters.
